This project appears to be a web application providing an AI tutor interface. It uses a Python Flask backend to interact with Ollama (for AI models and embeddings), a SQLite database for chat history, FAISS for vector storage (for RAG), and PyMuPDF/pdfplumber for PDF processing. The frontend is a simple HTML page with Bootstrap styling and JavaScript for dynamic interaction.

Here's the detailed breakdown:

**1. `backend/.env`**

* **Purpose:** This file stores environment-specific configuration variables, particularly sensitive ones like API URLs or paths, and allows easy switching of settings (like Ollama models or base URL) without changing the code directly.
* **Contents:**
  * `OLLAMA_BASE_URL`: Specifies the network address and port where the Ollama server is running. The comments show examples for different network setups (local IP, ngrok tunnel). The active line `https://64ab-106-215-170-223.ngrok-free.app/` indicates it's configured to use an ngrok tunnel URL.
  * `OLLAMA_MODEL`: Specifies the name of the Language Model (LLM) pulled in Ollama to be used for chat responses and analysis tasks. Examples like `deepseek-r1`, `llama3:8b`, `mistral:7b-instruct`, `qwen2.5:14b-instruct` are commented out. The active line `llama3.1:latest` sets the current model.
  * `OLLAMA_EMBED_MODEL`: Specifies the name of the embedding model pulled in Ollama to be used for converting text into numerical vectors for the vector store. Examples like `nomic-embed-text` are commented out. The active line `mxbai-embed-large` sets the current embedding model.
  * `OLLAMA_REQUEST_TIMEOUT`: An optional setting to define the timeout in seconds for requests made to the Ollama server. Commented out, meaning the default (likely 180 seconds from `config.py`) will be used.
  * `FAISS_FOLDER`: Specifies the directory name (relative to the backend) where the FAISS vector index will be stored. Commented out, defaults to `faiss_store`.
  * `UPLOAD_FOLDER`: Specifies the directory name (relative to the backend) where uploaded PDF files will be saved. Commented out, defaults to `uploads`.
  * `DATABASE_NAME`: Specifies the filename for the SQLite database. Commented out, defaults to `chat_history.db`.
  * `DEFAULT_PDFS_FOLDER`: Specifies the directory name (relative to the backend) where default PDF files are stored for initial indexing. Commented out, defaults to `default_pdfs`.
  * `RAG_CHUNK_K`: Number of top *unique* chunks retrieved from the vector store to be included in the context sent to the LLM for generating a chat response. Commented out, defaults to 5.
  * `RAG_SEARCH_K_PER_QUERY`: Number of chunks to retrieve from the vector store for *each* sub-query generated by the LLM before deduplication. Commented out, defaults to 3.
  * `MULTI_QUERY_COUNT`: Number of sub-queries the LLM will generate from the user's original query during the RAG process. Set to 0 to disable multi-query RAG. Commented out, defaults to 3.
  * `ANALYSIS_MAX_CONTEXT_LENGTH`: Maximum number of characters from a document's text that will be sent to the LLM for analysis tasks (FAQ, Topics, Mindmap). Commented out, defaults to 8000.
  * `LOGGING_LEVEL`: Sets the minimum severity level for logs to be recorded (DEBUG, INFO, WARNING, ERROR, CRITICAL). Commented out, defaults to INFO.

**2. `backend/config.py`**

* **Purpose:** This file loads the environment variables from `.env` and defines application-wide configuration constants, including paths, default values, and prompt templates used by the AI models. It also sets up the logging configuration.
* **Contents:**
  * **Imports:** Imports `os`, `dotenv`, `logging`, `PromptTemplate` from `langchain.prompts`.
  * `dotenv.load_dotenv(dotenv_path=...)`: Explicitly loads variables from the `.env` file located in the same directory as `config.py`.
  * **Environment Variables & Defaults:** Reads environment variables using `os.getenv()`, providing default values if the variable is not set in `.env` or the environment. It constructs absolute paths for folders (`FAISS_FOLDER`, `UPLOAD_FOLDER`, `DATABASE_PATH`, `DEFAULT_PDFS_FOLDER`) relative to the `backend` directory.
  * `ALLOWED_EXTENSIONS`: A set containing allowed file extensions for uploads (`{'pdf'}`).
  * `LOGGING_LEVEL_NAME`, `LOGGING_LEVEL`, `LOGGING_FORMAT`: Defines logging severity level and format based on the `LOGGING_LEVEL` environment variable.
  * **Prompt Templates:** Defines `PromptTemplate` objects using `langchain.prompts`. These templates structure the input sent to the LLM for specific tasks.
    * `SUB_QUERY_PROMPT_TEMPLATE`: Used by the RAG process to ask the LLM to break down a user query into multiple search queries. It takes `query` and `num_queries` as input variables. Instructions emphasize outputting *only* the queries, one per line.
    * `SYNTHESIS_PROMPT_TEMPLATE`: Used by the RAG process to ask the LLM to synthesize a final answer based on the original `query` and the retrieved `context`. It includes detailed instructions for the LLM, *mandating* a `<thinking>...</thinking>` block for reasoning before providing the final answer, prioritizing context, citing sources `[N]`, integrating general knowledge, and using Markdown.
    * `_ANALYSIS_THINKING_PREFIX`: A helper string containing common instructions for analysis prompts, including the recommended thinking process and document text delimiters.
    * `ANALYSIS_PROMPTS`: A dictionary mapping analysis types (`"faq"`, `"topics"`, `"mindmap"`) to their specific `PromptTemplate`. Each template includes the `_ANALYSIS_THINKING_PREFIX` and task-specific instructions (e.g., format for FAQ, topics list, Mermaid code block).
  * `setup_logging()`: A function to configure Python's logging based on the defined level and format. It also sets higher logging levels for noisy libraries (`urllib3`, `httpx`, `faiss.loader`, `pdfminer`) to reduce log spam. Logs key configuration values at DEBUG level.

**3. `backend/utils.py`**

* **Purpose:** Contains utility functions used across the backend for tasks like file validation, parsing LLM responses, and extracting information from responses.
* **Contents:**
  * **Imports:** Imports `re`, `logging`, `json`, `os`, `ALLOWED_EXTENSIONS` from `config`.
  * `allowed_file(filename)`: Checks if a given `filename` has an extension present in the `ALLOWED_EXTENSIONS` set defined in `config.py`.
  * `parse_llm_response(full_response)`: Takes the raw text response from an LLM. It uses a regular expression (`re.compile`) to find and extract content enclosed within `<thinking>...</thinking>` tags (case-insensitive, allowing attributes in the opening tag). It returns a tuple: the text *outside* the thinking block (the `user_answer`) and the text *inside* the thinking block (`thinking_content`), or `None` if no thinking block is found. It handles potential whitespace around the tags and ensures the user answer is stripped of the thinking block.
  * `extract_references(answer_text, context_docs_map)`: Parses the `answer_text` to find citation markers formatted as `[N]` (where N is an integer). It uses the `context_docs_map` (a dictionary mapping the integer citation number to document details like source filename and content) to retrieve the source information for each cited number. It deduplicates references based on the source filename, storing the first citation number found for that source. It creates a list of dictionaries, each containing the citation `number`, `source` filename, and a `content_preview` (truncated and cleaned from the original chunk content). The resulting list is sorted by source filename.
  * `escape_html(unsafe_str)`: Performs basic HTML escaping on a string to make it safe for display in an HTML context, replacing characters like `<`, `>`, `&`, `"`, `'` with their HTML entities. Handles `None` input and attempts to convert non-string inputs.

**4. `backend/database.py`**

* **Purpose:** Manages interactions with the SQLite database, specifically for storing and retrieving chat history messages.
* **Contents:**
  * **Imports:** Imports `sqlite3`, `logging`, `json`, `uuid`, `datetime`, `timezone`, `DATABASE_PATH` from `config`.
  * `get_db_connection()`: Establishes a connection to the SQLite database specified by `DATABASE_PATH`. It configures the connection to return rows as dictionary-like objects (`sqlite3.Row`), enables Write-Ahead Logging (WAL) for better concurrency, sets a busy timeout, and enforces foreign key constraints. Includes error handling for connection failures.
  * `init_db()`: Creates the `messages` table if it doesn't exist. The table schema includes `message_id` (UUID, primary key), `session_id`, `sender` ('user' or 'bot' with a check constraint), `message_text`, `timestamp` (stored as UTC ISO8601 text with a default value), `references_json` (text for JSON string), and `cot_reasoning` (text for thinking content). It also includes simple schema migration logic to add `references_json`, `cot_reasoning`, and `timestamp` columns if they are missing from an existing table. An index is created on `session_id` and `timestamp` for faster history retrieval.
  * `save_message(session_id, sender, message_text, references=None, cot_reasoning=None)`: Inserts a new message into the `messages` table. It generates a UUID for `message_id`. It serializes the `references` list/dict to a JSON string if provided. It saves the `session_id`, `sender`, `message_text`, JSON references, and `cot_reasoning` (thinking content). Returns the `message_id` on success, `None` on failure (logging the error).
  * `get_messages_by_session(session_id)`: Retrieves all messages associated with a given `session_id`, ordered by `timestamp`. It fetches all columns, including `references_json` and `cot_reasoning`. For each retrieved row, it converts it to a dictionary, safely parses the `references_json` string back into a Python list/dict, renames `cot_reasoning` to `thinking`, and ensures the timestamp is included. Returns a list of message dictionaries, or `None` if a database error occurs. Returns an empty list if no messages are found for the session ID.

**5. `backend/ai_core.py`**

* **Purpose:** Contains the core logic for interacting with Ollama (embeddings and LLM), processing documents, building/loading/saving the vector store, and performing RAG and analysis tasks.
* **Contents:**
  * **Imports:** Imports necessary modules from `os`, `logging`, `fitz` (commented out, but `pdfplumber` is used), `re`, `langchain_community.vectorstores`, `langchain_ollama`, `langchain.text_splitter`, `langchain.docstore.document`, `langchain.chains`, `langchain.prompts`, `config`, `utils`, `pdfplumber`.
  * **Global State:** Declares `document_texts_cache` (dict to store extracted text by filename), `vector_store` (FAISS index object), `embeddings` (OllamaEmbeddings instance), `llm` (ChatOllama instance) as global variables to be managed within functions.
  * `initialize_ai_components()`: Initializes the global `embeddings` (`OllamaEmbeddings`) and `llm` (`ChatOllama`) instances using configuration from `config.py`. Includes basic test calls (`embed_query`, `invoke`) to verify connectivity and model availability. Handles import errors (if Langchain packages are missing) and general exceptions (e.g., connection errors to Ollama). Returns the initialized objects or `(None, None)` on failure.
  * `load_vector_store()`: Attempts to load the FAISS index from the `FAISS_FOLDER`. Requires the global `embeddings` instance to be initialized. Uses `FAISS.load_local` with `allow_dangerous_deserialization=True`. Sets the global `vector_store`. Returns `True` if loaded successfully (even if empty), `False` otherwise (file not found, corrupted, or error).
  * `save_vector_store()`: Saves the current global `vector_store` to the `FAISS_FOLDER`. Creates the directory if it doesn't exist. Returns `True` on success, `False` on failure or if `vector_store` is `None`.
  * `load_all_document_texts()`: Iterates through `DEFAULT_PDFS_FOLDER` and `UPLOAD_FOLDER`, calls `extract_text_from_pdf` for each PDF, and stores the extracted text in the global `document_texts_cache` dictionary, keyed by filename. Resets the cache before loading.
  * `extract_text_from_pdf(pdf_path)`: Uses `pdfplumber` to open a PDF file and extract text page by page. Includes basic text cleaning (whitespace, newlines). Returns the combined, cleaned text string or `None` if the file is not found, empty, or extraction fails.
  * `create_chunks_from_text(text, filename)`: Takes a text string and filename. Uses `RecursiveCharacterTextSplitter` with configured `chunk_size` and `chunk_overlap` to split the text into smaller segments. Creates LangChain `Document` objects for each chunk, adding `source` (filename) and `chunk_index` metadata. Returns a list of these `Document` objects.
  * `add_documents_to_vector_store(documents)`: Adds a list of `Document` objects to the global `vector_store`. If `vector_store` is `None`, it creates a new FAISS index using `FAISS.from_documents`. If `vector_store` exists, it uses `vector_store.add_documents`. After adding, it calls `save_vector_store` to persist the changes. Returns `True` if addition and saving are successful, `False` otherwise.
  * `generate_sub_queries(query)`: Uses the global `llm` and the `SUB_QUERY_PROMPT_TEMPLATE` from `config` to generate multiple search queries from the user's original `query`. Uses Langchain's `LLMChain`. Returns a list of generated queries, always including the original query and ensuring the list size respects `MULTI_QUERY_COUNT`. Returns just the original query if LLM is not initialized or `MULTI_QUERY_COUNT` is 0.
  * `perform_rag_search(query)`: Orchestrates the RAG retrieval process. Calls `generate_sub_queries`. Performs `vector_store.similarity_search_with_score` for each query, retrieving up to `RAG_SEARCH_K_PER_QUERY` chunks per query. Deduplicates the retrieved chunks based on source and chunk index, keeping the best score for duplicates. Selects the top `RAG_CHUNK_K` unique chunks. Formats these chunks into a single string (`formatted_context_text`) for the LLM prompt, including `[N]` citations and source/chunk info. Creates `context_docs_map` mapping citation numbers to document details. Returns the list of `Document` objects, the formatted context text, and the `context_docs_map`. Handles cases where the vector store is not ready or empty, or search errors occur.
  * `synthesize_chat_response(query, context_text)`: Uses the global `llm` and the `SYNTHESIS_PROMPT_TEMPLATE` from `config` to generate the final chat response. It passes the user `query` and the `context_text` (from RAG) to the LLM using `llm.invoke()`. It then calls `parse_llm_response` from `utils` to separate the user-facing answer and the thinking content. Returns a tuple `(user_answer, thinking_content)`. Handles LLM invocation errors and prompt formatting errors.
  * `generate_document_analysis(filename, analysis_type)`: Generates analysis (FAQ, Topics, Mindmap) for a specific document. Retrieves the document text from the `document_texts_cache` first; if not found, it attempts to load and extract it from the upload or default folders. Truncates the text to `ANALYSIS_MAX_CONTEXT_LENGTH` if necessary. Retrieves the appropriate prompt template from `ANALYSIS_PROMPTS` based on `analysis_type`. Calls the global `llm` using `llm.invoke()` with the formatted prompt. Calls `parse_llm_response` to separate the analysis content and thinking content. Returns a tuple `(analysis_content, thinking_content)`. Returns error messages and potentially thinking content on failure (file not found, extraction failed, invalid analysis type, LLM error).

**6. `backend/default.py`**

* **Purpose:** This is a standalone script (intended to be run manually or via a startup process, separate from the Flask app) to perform the initial setup: check the Ollama connection and build or update the FAISS vector index from the PDF files located in the `DEFAULT_PDFS_FOLDER`.
* **Contents:**
  * **Imports:** Imports `os`, `logging`, `sys`, `requests`, and configuration/AI core functions/globals from `config` and `ai_core`.
  * `setup_logging()`: Calls the logging setup from `config.py`.
  * `check_ollama_connection(base_url, timeout=5)`: Attempts to connect to the Ollama server's `/api/tags` endpoint to verify it's running and accessible. Returns `True` on success, `False` on connection errors, timeouts, or invalid responses.
  * `get_existing_sources_from_index(vs)`: Inspects the loaded FAISS vector store (`vs`) to find the source filenames present in the metadata of the stored documents. This is used to avoid re-processing PDFs that are already indexed. Returns a set of filenames. Includes error handling for unexpected docstore structures.
  * `build_initial_faiss_index()`: The main function of the script.
    * Checks if `DEFAULT_PDFS_FOLDER` exists.
    * Performs an Ollama connection pre-check using `check_ollama_connection`. Exits if Ollama is unreachable.
    * Initializes AI components (`embeddings`, `llm`) using `ai_core.initialize_ai_components`. Exits if initialization fails.
    * Loads any existing FAISS index using `ai_core.load_vector_store`. Retrieves existing source filenames using `get_existing_sources_from_index`.
    * Lists all PDF files in `DEFAULT_PDFS_FOLDER`.
    * Identifies PDFs that are *not* in the set of existing filenames.
    * Iterates through the new PDFs, calls `ai_core.extract_text_from_pdf` and `ai_core.create_chunks_from_text`. Collects all new document chunks.
    * If new chunks are found, calls `ai_core.add_documents_to_vector_store` to add them to the index (creating it if necessary) and save it.
    * Logs the process and reports success or failure.
  * `if __name__ == "__main__":`: Executes `build_initial_faiss_index()` when the script is run directly and exits with a status code (0 for success, 1 for expected failure, 2 for unexpected error).

**9. `backend/app.py`**

* **Purpose:** The main Flask application file. It sets up the web server, defines routes for handling HTTP requests (serving HTML, getting status, listing documents, uploading, analyzing, chatting, getting history), and orchestrates the use of other backend modules (`config`, `database`, `ai_core`, `utils`).
* **Contents:**
  * **Imports:** Imports necessary modules from `os`, `logging`, `json`, `uuid`, `flask`, `flask_cors`, `werkzeug.utils`, `waitress`, `datetime`, `timezone`. Imports `config`, `database`, `ai_core`, `utils`.
  * `config.setup_logging()`: Calls the logging setup function from `config.py` early.
  * **Flask App Setup:** Initializes the Flask app, specifying template and static folders.
  * **CORS Configuration:** Configures Flask-CORS to allow requests from all origins (`*`). **Note:** This is explicitly logged as insecure for public deployment but suitable for development/LAN.
  * **Upload Configuration:** Sets the upload folder path from `config.UPLOAD_FOLDER` and defines a `MAX_CONTENT_LENGTH`. Ensures the upload directory exists.
  * **Application Initialization Flags:** Global boolean flags (`app_db_ready`, `app_ai_ready`, `app_vector_store_ready`, `app_doc_cache_loaded`) to track the status of core components after initialization.
  * `initialize_app()`: A function that performs the core backend initialization steps: calls `database.init_db()`, `ai_core.initialize_ai_components()`, `ai_core.load_vector_store()`, and `ai_core.load_all_document_texts()`. It sets the global status flags based on the success of these steps. It includes logging for each step and overall status. This function is designed to run only once.
  * `@app.before_request`: A Flask decorator that ensures the `initialize_app` function is called before the first request is handled by the application instance.
  * **Flask Routes (`@app.route`)**:
    * `/`: Renders the `index.html` template.
    * `/favicon.ico`: Returns a 204 No Content response, common practice for favicon requests.
    * `/status` (GET): Returns a JSON response indicating the readiness status of the database, AI components, vector store (including entry count), and document cache. Includes a timestamp.
    * `/documents` (GET): Lists PDF files found in the default and upload folders using `os.listdir` and basic filtering. Returns a JSON response with separate lists for `default_files` and `uploaded_files`, and any `errors` encountered during listing.
    * `/upload` (POST): Handles file uploads. Checks if the AI components (specifically embeddings) are ready (`app_ai_ready`). Validates the request and file type (`utils.allowed_file`). Secures the filename. Saves the file to the `UPLOAD_FOLDER`. Calls `ai_core.extract_text_from_pdf`, `ai_core.create_chunks_from_text`, and `ai_core.add_documents_to_vector_store`. Updates the `ai_core.document_texts_cache`. Returns a JSON success message with filename and new vector count, or a JSON error message with an appropriate HTTP status code (400 for bad request/file, 503 for service unavailable, 500 for internal processing errors). Includes cleanup logic to remove the file if processing fails.
    * `/analyze` (POST): Handles analysis requests. Checks if the LLM is ready (`app_ai_ready`). Parses `filename` and `analysis_type` from the JSON request body. Validates the input. Calls `ai_core.generate_document_analysis` with the filename and type. Returns a JSON response containing the `content` and `thinking` from the analysis, or a JSON error message with an appropriate HTTP status code (400 for bad request, 404 for file not found, 503 for AI unavailable, 500 for internal errors).
    * `/chat` (POST): Handles chat requests. Checks if database (`app_db_ready`) and AI components (`app_ai_ready`) are ready. Parses `query` and `session_id` from the JSON request. Manages the session ID (generates a new one if none provided or invalid). Saves the user message to the database using `database.save_message`. Performs the RAG pipeline: calls `ai_core.perform_rag_search` (if vector store is ready and RAG enabled), then `ai_core.synthesize_chat_response`. Calls `utils.extract_references` to find citations in the bot's answer. Saves the bot's response (answer, references, thinking) to the database. Returns a JSON response with the `answer`, `session_id`, `references`, and `thinking`. Includes extensive error handling throughout the pipeline, returning appropriate error messages and status codes (503 for service unavailable, 500 for internal errors) and attempting to log errors to the database.
    * `/history` (GET): Retrieves chat history for a given `session_id` from the database. Checks if the database is ready. Parses and validates the `session_id` from query parameters. Calls `database.get_messages_by_session`. Returns a JSON list of message dictionaries, or a JSON error message with an appropriate status code (400 for invalid ID, 503 for DB unavailable, 500 for internal errors).
  * `if __name__ == '__main__':`: This block runs when `app.py` is executed directly (e.g., `python app.py`). It calls `initialize_app()` to ensure setup runs. It determines the host (`0.0.0.0` for LAN access) and port (from environment or defaults to 5000). It then uses `waitress.serve` to run the Flask application using a production-ready WSGI server, logging startup information and component status.
